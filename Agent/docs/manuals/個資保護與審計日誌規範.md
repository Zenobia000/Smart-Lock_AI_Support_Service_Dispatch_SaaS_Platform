# 個資保護與審計日誌規範

本文件說明系統如何自動擷取與管理使用者個資、資料流向與儲存邊界、審計日誌的記錄規範，以及安全強化建議。

---

## 1. 個資自動擷取機制

### 1.1 Regex 設定來源

系統透過正則表達式（Regex）從對話中自動擷取使用者的電話號碼與地址。Regex 的設定來源如下：

1. **主要來源**：`config.toml` 的 `[user_profile.extraction]` 區塊

   ```toml
   [user_profile.extraction]
   phone_regex   = '09\d{2}[\-\s]?\d{3}[\-\s]?\d{3}'
   address_regex = '[\u4e00-\u9fff]*(?:市|縣)...'
   ```

2. **載入機制**：`core/constants.py` 在模組初始化時從 `EXTRACTION_CONFIG` 動態載入，編譯為 `PHONE_REGEX` 與 `ADDRESS_REGEX` 兩個 `re.compile()` 物件

3. **Fallback**：若 `config.toml` 未設定，`core/constants.py` 使用台灣格式作為硬編碼 fallback

> **TOML 語法提示**：Regex 字串請使用 TOML 的 literal string（單引號 `'...'`），此格式不轉義反斜線，避免 `\d` 被誤解為轉義序列。

### 1.2 擷取流程

系統在兩個環節使用個資擷取：

**環節一：update_profile 節點**

- 時機：每輪對話結束後（`merge_answers` 之後）
- 方式：LLM 根據 `update_profile.md` prompt 從對話中萃取結構化個資
- 輸出：寫入 `data/profiles/{user_id}.md`
- 擷取欄位：設備型號、品牌、居住環境、聯絡地址、聯絡電話等

**環節二：transfer_to_human 工具**

- 時機：Agent 判斷需要轉接真人客服時呼叫
- 方式：使用 `PHONE_REGEX` / `ADDRESS_REGEX` 從使用者輪廓與當前問題中提取電話與地址
- 輸出：填入轉接表單（模板為 `agents/prompts/transfer_human_form.md`）
- 目的：讓真人客服接手時已具備基本聯絡資訊

### 1.3 國際化擴充

系統的個資擷取 Regex 完全配置化，支援國際化擴充：

- 修改 `config.toml` 的 `[user_profile.extraction]` 中的 `phone_regex` 與 `address_regex` 即可
- 無需修改任何 Python 程式碼
- `core/constants.py` 會在啟動時自動載入新的 Regex

例如，若需支援日本電話格式：

```toml
[user_profile.extraction]
phone_regex = '0[0-9]{1,4}[\-\s]?[0-9]{1,4}[\-\s]?[0-9]{3,4}'
```

## 2. 資料流向與儲存邊界

### 2.1 留在本地的資料（不離開伺服器）

| 資料 | 檔案路徑 | 內容 |
|------|---------|------|
| 審計日誌 | `data/db/audit_log.db` | 完整對話原文（user_raw / user / ai） |
| 對話記憶 | `data/db/chat_history.db` | LangGraph checkpoint 狀態與摘要 |
| 使用者輪廓 | `data/profiles/*.md` | 使用者個資（設備、地址、電話） |
| 向量資料庫 | `data/db/chroma_db_*` | 產品手冊與故障排除文件的向量索引 |

> 上述資料均儲存於伺服器本地磁碟，不經由網路傳輸。

### 2.2 傳送至雲端 LLM 的資料（Gemini / Vertex AI）

當 `[llm].provider` 設定為 `gemini` 或 `vertexai` 時，以下資料會傳送至 Google 雲端：

- 對話訊息（包含使用者問題與歷史對話）
- 使用者輪廓文字（由 `pre_process` 注入）
- System Prompt（Router / Agent / Summarizer 等模板內容）
- 檢索器回傳的文件片段

> **注意**：若 `[llm].provider = "ollama"`，所有 LLM 推論均在本地執行，資料不離開伺服器。

### 2.3 傳送至外部 API 的資料

| 外部服務 | 傳送內容 | 傳送方式 |
|---------|---------|---------|
| Order API（`db_order_api`） | 僅 `keyword` 查詢參數（使用者問題關鍵字） | HTTP GET/POST，可附帶 Bearer Token |
| DuckDuckGo（`db_web_search`） | 僅搜尋關鍵字 | HTTPS 查詢 |

兩者均不傳送使用者個資（電話、地址）或完整對話歷史。

## 3. 審計日誌規範

### 3.1 記錄時機與角色

系統在三個時機點記錄審計日誌，各自對應不同的 `role` 值：

| role | 記錄時機 | 說明 |
|------|---------|------|
| `user_raw` | webhook 收到每條訊息時立即記錄 | debounce 之前，確保原始碎裂訊息不遺失 |
| `user` | debounce 結束後，送入 LangGraph 前 | 合併後的完整使用者訊息 |
| `ai` | LangGraph 執行完畢後 | AI 回覆的完整內容 |

**範例**：使用者連發 3 條訊息，預期產生 5 筆紀錄：

- 3 筆 `user_raw`（每條原始訊息各一筆）
- 1 筆 `user`（3 條訊息合併後）
- 1 筆 `ai`（AI 回覆）

### 3.2 資料庫 Schema

```sql
CREATE TABLE audit_log (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id   TEXT NOT NULL,
    role      TEXT NOT NULL,      -- 'user_raw' | 'user' | 'ai'
    content   TEXT NOT NULL,
    timestamp TEXT NOT NULL       -- UTC ISO 8601 格式
);
```

| 欄位 | 類型 | 說明 |
|------|------|------|
| `id` | INTEGER | 自增主鍵 |
| `user_id` | TEXT | LINE 使用者 ID |
| `role` | TEXT | 訊息角色（`user_raw` / `user` / `ai`） |
| `content` | TEXT | 訊息原文 |
| `timestamp` | TEXT | UTC 時間戳，ISO 8601 格式（如 `2026-03-11T08:30:00+00:00`） |

### 3.3 完整性保證

- **先寫後抖**：`user_raw` 在 webhook 迴圈中立即寫入，先於 debounce 計時器啟動，確保即使系統中斷也不遺失原始訊息
- **靜默容錯**：若 startup 初始化 `audit_storage` 失敗，系統會將其設為 `None` 並靜默跳過日誌寫入，不阻斷主服務流程
- **UTC 時間戳**：所有 `timestamp` 使用 `datetime.now(timezone.utc).isoformat()`，確保跨時區一致性

## 4. 安全強化建議

### 4.1 環境變數管理

- 所有機密資訊透過 `_env` 慣例從環境變數讀取，不寫入 `config.toml`
- `.env` 檔案已加入 `.gitignore`，確保不進入版本控制
- 生產環境建議使用雲端 Secret Manager（Google Secret Manager / AWS Secrets Manager / Azure Key Vault）取代 `.env` 檔案

### 4.2 資料庫安全

- **SQLite**：建議設定檔案權限為 `chmod 600`，僅允許服務帳號讀寫
- **PostgreSQL**：啟用 SSL/TLS 加密連線，限制連線來源 IP
- 定期備份並驗證備份的可還原性

### 4.3 API Key 定期輪替

建議定期輪替以下金鑰，降低洩漏風險：

| 金鑰 | 建議輪替頻率 | 輪替方式 |
|------|------------|---------|
| `GEMINI_API_KEY` | 每 90 天 | Google AI Studio 重新產生 |
| `LINE_CHANNEL_ACCESS_TOKEN` | 每 90 天 | LINE Developer Console 重新發行 |
| `LINE_CHANNEL_SECRET` | 視需要 | 重新建立 Channel（影響較大，非必要不輪替） |
| `ORDER_API_TOKEN` | 依 API 供應商政策 | 聯繫 API 供應商 |

### 4.4 生產環境強化

- **移除 `--reload`**：開發用旗標，生產環境務必移除，避免程式碼變更觸發意外重啟
- **限制 webhook IP**：在 Nginx 或防火牆層限制 `/webhook` 端點僅接受 LINE 平台的 IP 範圍
- **CORS 設定**：FastAPI 預設不啟用 CORS，若有前端面板需求，應嚴格設定允許的 Origin
- **日誌脫敏**：生產環境的應用程式日誌（stdout/stderr）應避免印出使用者個資，可在 logging 層加入脫敏過濾器
- **請求頻率限制**：考慮在反向代理層加入 rate limiting，防止惡意大量請求
