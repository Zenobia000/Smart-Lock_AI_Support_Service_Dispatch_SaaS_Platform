# 系統擴充開發指南

> **[給 AI 助手的系統指令]**
> 你正在協助擴充一個基於 LangGraph 與 LangChain 建構的異步 (Async) 多 Agent 客服系統。
> 在生成任何程式碼之前，請務必詳細閱讀以下架構規範。**絕對禁止**為了新增單一功能而修改核心圖表邏輯 (`graph/builder.py`)。本系統嚴格遵循「設定檔驅動 (Config-Driven)」與「工廠模式 (Factory Pattern)」。

---

## 1. 架構總覽

```
config.toml  ─────────────────────────────────────────────────
  │
  ├── [llm]          → llms/__init__.py         → LLM 實例
  ├── [embedding]    → embeddings/__init__.py   → Embedding 實例（全域）
  ├── [[databases]]  → retrievers/__init__.py   → Tool 實例
  ├── [[agents]]     → agents/__init__.py       → Agent 子圖
  ├── [[intents]]    → graph/nodes.py (router)  → 意圖路由
  ├── [memory]       → memory/__init__.py       → Checkpointer
  ├── [user_profile] → profiles/manager.py      → 輪廓持久化
  ├── [user_profile.extraction] → core/constants.py → 個資提取 Regex
  ├── [storage]      → storage/__init__.py      → 審計日誌
  ├── [prompts]      → graph/nodes.py           → Prompt 路徑集中管理
  └── [required_slots] → agents/__init__.py     → Slot Filling
```

### 管線流程

```
START → pre_process → manage_memory → router
  → [Send() fan-out] → agent 子圖 / out_of_domain / human
  → merge_answers → update_profile → post_process → END
```

### 模組工廠一覽

| 資料夾 | 職責 | 註冊方式 | 介面要求 |
|--------|------|---------|---------|
| `llms/` | 語言模型 | `LLM_REGISTRY` dict | 回傳 `BaseChatModel` |
| `embeddings/` | 向量模型 | `REGISTRY` dict | 回傳 `Embeddings` |
| `retrievers/` | 檢索器 | `REGISTRY` dict | 繼承 `BaseRetriever` |
| `memory/` | 對話記憶 | `get_checkpointer()` elif | 回傳 `BaseCheckpointSaver` |
| `storage/` | 審計日誌 | `STORAGE_REGISTRY` dict | `SqliteAuditStorage` |
| `agents/` | Agent 子圖 | `config.toml [[agents]]` | 自動建構 |
| `tools/` | LangChain 工具 | 自動從 `[[databases]]` 產生 | `StructuredTool` |
| `profiles/` | 使用者輪廓 | `config.toml [user_profile]` | Markdown 檔案 |

## 2. 開發鐵則 (Golden Rules)

1. **從不硬編碼**：API Key、模型名稱、連線網址等，一律從 `config: dict` 或 `os.getenv` 讀取。機密值使用 `_env` 後綴欄位指向 `.env` 變數名。
2. **單一職責**：新增一個供應商（OpenAI、Anthropic、Qdrant...），建立一個全新的 `.py` 檔案。
3. **註冊表機制**：寫完 `.py` 後，**只修改**該模組的 `__init__.py`，將新功能加入 `REGISTRY` 或判斷式。
4. **介面相容**：回傳物件必須相容 LangChain / LangGraph 標準介面。
5. **不碰核心**：不修改 `graph/builder.py`、`graph/nodes.py`、`graph/state.py`。所有擴充透過 config + 工廠完成。

## 3. 新增語言模型 (LLM)

### 位置與介面

| 項目 | 說明 |
|------|------|
| 目標資料夾 | `llms/` |
| 檔名慣例 | `{provider}_model.py` |
| 函數命名 | `build_{provider}_llm(config: dict)` |
| 回傳類型 | `BaseChatModel`（如 `ChatOpenAI`, `ChatAnthropic`） |
| 註冊位置 | `llms/__init__.py` → `LLM_REGISTRY` |

### 現有供應商

| Provider | 檔案 | 設定重點 |
|----------|------|---------|
| `ollama` | `ollama_model.py` | `base_url`、`model_name`、`temperature` |

| `vertexai` | `vertexai_model.py` | `project_id_env`、`location_env`、`model_name` |

### 擴充範例：新增 OpenAI

**Step 1** — 建立 `llms/openai_model.py`

```python
import os
from langchain_openai import ChatOpenAI

def build_openai_llm(config: dict):
    api_key = os.getenv(config.get("api_key_env", "OPENAI_API_KEY"))
    return ChatOpenAI(
        model=config.get("model_name", "gpt-4o"),
        temperature=config.get("temperature", 0.3),

**Step 2** — 修改 `llms/__init__.py`

from .openai_model import build_openai_llm

LLM_REGISTRY = {
    "ollama": build_ollama_llm,
    "gemini": build_gemini_llm,
    "vertexai": build_vertexai_llm,
    "openai": build_openai_llm,        # ← 新增

```

**Step 3** — 修改 `config.toml`

```toml
[llm]
provider = "openai"
model_name = "gpt-4o"
temperature = 0.3
api_key_env = "OPENAI_API_KEY"
```

**Step 4** — `pip install langchain-openai`

## 4. 新增檢索器 / 資料庫 (Retriever)

### 位置與介面

| 項目 | 說明 |
|------|------|
| 目標資料夾 | `retrievers/` |
| 檔名慣例 | `{provider}_store.py` |
| 基底類別 | **必須繼承** `from .base import BaseRetriever` |
| 必實作方法 | `def setup(self)` + `async def aretrieve(self, question: str) -> str` |
| 註冊位置 | `retrievers/__init__.py` → `REGISTRY` |

### BaseRetriever 介面

```python
class BaseRetriever(ABC):
    def __init__(self, config: dict):
        self.config = config
        self.setup()

    @abstractmethod
    def setup(self):
        """初始化：從 self.config 讀取設定、建立連線"""

    @abstractmethod
    async def aretrieve(self, question: str) -> str:
        """非同步檢索，回傳格式化字串。查無資料時回傳預設提示。"""
```

### 現有檢索器

| Type | 檔案 | 用途 | 主要設定 |
|------|------|------|---------|
| `chroma` | `chroma_store.py` | 向量資料庫 | `path`、`top_k`（Embedding 使用全域 `[embedding]` 設定） |
| `api` | `api_store.py` | RESTful API 查詢 | `endpoint_env`、`method`、`query_param`、`response_key` |
| `web_search` | `web_search.py` | 網頁搜尋 | `search_engine`、`max_results` |

### 擴充範例：新增 Qdrant

**Step 1** — 建立 `retrievers/qdrant_store.py`

```python
import os
from qdrant_client import AsyncQdrantClient
from .base import BaseRetriever
from embeddings import get_embedding

class QdrantRetriever(BaseRetriever):
    def setup(self):
        self.url = os.getenv(self.config.get("url_env")) or self.config.get("url")
        self.collection = self.config.get("collection_name")
        self.top_k = self.config.get("top_k", 3)
        self.client = AsyncQdrantClient(url=self.url)
        self.embedding = get_embedding(self.config)

    async def aretrieve(self, question: str) -> str:
        vector = self.embedding.embed_query(question)
        results = await self.client.search(
            collection_name=self.collection,
            query_vector=vector,
            limit=self.top_k,
        )
        if not results:
            return "查無相關資料。"
        return "\n---\n".join([r.payload.get("text", "") for r in results])
```

**Step 2** — 修改 `retrievers/__init__.py`

```python
from .qdrant_store import QdrantRetriever

REGISTRY = {
    "chroma": ChromaRetriever,
    "api": APIStoreRetriever,
    "web_search": WebSearchRetriever,
    "qdrant": QdrantRetriever,          # ← 新增
}
```

**Step 3** — 修改 `config.toml`

```toml
[[databases]]
name = "db_product_faq"
type = "qdrant"
description = "產品 FAQ 知識庫"
url = "http://localhost:6333"
url_env = "QDRANT_URL"
collection_name = "product_faq"
top_k = 3
# Embedding 自動使用全域 [embedding] 設定，無需重複定義
```

> 新增的 `[[databases]]` 會自動被 `tools/__init__.py` 包裝為 `StructuredTool`，可直接在 `[[agents]]` 的 `tools` 中引用。

## 5. 新增向量模型 (Embedding)

### 位置與介面

| 項目 | 說明 |
|------|------|
| 目標資料夾 | `embeddings/` |
| 檔名慣例 | `{provider}_embed.py` |
| 函數命名 | `build_{provider}_embedding(config: dict)` |
| 回傳類型 | LangChain `Embeddings` 介面物件 |
| 註冊位置 | `embeddings/__init__.py` → `REGISTRY` |

### 全域 Embedding 設定

Embedding 模型透過 `config.toml` 的 `[embedding]` 區塊全域設定，所有使用向量檢索的 Retriever（如 ChromaDB）會自動套用此設定，無需在 `[[databases]]` 中重複定義。

```toml
[embedding]
provider     = "ollama"
model        = "nomic-embed-text"
base_url     = "http://localhost:11434"
base_url_env = "OLLAMA_BASE_URL"
```

`get_embedding()` 函數在未收到帶有 `embedding_provider` 的 config 時，會自動 fallback 至全域 `EMBEDDING_CONFIG`。

### 現有供應商

| Provider | 檔案 | 設定重點 |
|----------|------|---------|
| `ollama` | `ollama_embed.py` | `model`、`base_url_env` |

### 擴充範例：新增 OpenAI Embedding

**Step 1** — 建立 `embeddings/openai_embed.py`

def build_openai_embedding(config: dict):

## 7. 新增 Agent（純設定，零程式碼）

Agent 是本系統最常見的擴充操作。**只需要修改 `config.toml` + 新增 prompt 檔案**，不需要寫任何 Python。

### 步驟

#### Step 1 — 確認工具已存在

Agent 使用的工具來自 `[[databases]]` 定義的檢索器 + 內建的 `transfer_to_human`。如果需要新的資料源，先按照 §4 新增 Retriever。

#### Step 2 — 撰寫 Agent Prompt

在 `agents/prompts/` 建立 `.md` 檔案，使用以下模板變數：

| 變數 | 來源 | 說明 |
|------|------|------|
| `{domain}` | `config.toml [system].domain` | 系統服務領域 |
| `{user_profile}` | 當前使用者輪廓 | 個人化上下文 |
| `{slots_section}` | `config.toml [required_slots]` | 自動產生的資訊收集提示 |

**Prompt 範本**（`agents/prompts/installer.md`）：

```markdown
你是一位專精「{domain}」安裝服務的客服專員。

## 使用者背景
{user_profile}

## 行為準則
1. 一律使用繁體中文回答。
2. 嚴格根據工具回傳的資料來回答，不可編造資訊。
3. 你必須使用工具來檢索資料，不可直接憑記憶回答。對話摘要（[前情提要]）僅供理解上下文，不可取代工具檢索——每次回覆前都必須先呼叫工具。
4. 語氣親切自然，像真人對話。

## 訊息完整性判斷
如果使用者的訊息不夠清楚或不完整，主動用友善的語氣詢問更多細節，而非直接檢索。
如果是問候語（「你好」「嗨」），友善回應並詢問需要什麼協助。

## 資訊不足時的處理
如果工具回傳的資料不足以完整回答：
1. 先提供你能找到的部分資訊
2. 嘗試用不同關鍵字或角度重新檢索
3. 根據你的專業知識提供通用建議或常見做法
4. 建議使用者提供更具體的資訊以便進一步協助
5. 只有當使用者明確表示以上方式都無法滿足需求時，才詢問是否需要轉接真人客服
不要主動提議轉接真人。不要只說「找不到資料」就結束對話。
{slots_section}
```

> **重要**：規則 3 必須包含「對話摘要（[前情提要]）僅供理解上下文，不可取代工具檢索」，否則 LLM 可能跳過工具呼叫。

#### Step 3 — 在 config.toml 註冊 Agent

```toml
[[agents]]
name = "installer"
label = "安裝服務專員"
description = "負責回答電子鎖安裝流程、到府安裝預約、安裝注意事項"
tools = ["db_installation_guide", "transfer_to_human"]
prompt_file = "agents/prompts/installer.md"
```

| 欄位 | 必填 | 說明 |
|------|------|------|
| `name` | ✅ | Agent 唯一識別名（英文，用於路由） |
| `label` | ✅ | 中文顯示名稱 |
| `description` | ✅ | Agent 職責描述（用於 Router 判斷） |
| `tools` | ✅ | 可用工具名稱清單（對應 `[[databases]].name` 或 `transfer_to_human`） |
| `prompt_file` | ✅ | Prompt 模板檔案路徑 |

#### Step 4 — 新增對應意圖

```toml
[[intents]]
name = "installation"
label = "安裝服務"
description = "使用者詢問安裝流程、到府安裝預約、安裝注意事項"
target = "installer"
require_slots = false
```

完成！系統會自動：
1. `build_all_agents()` 為新 agent 建構 `agent_llm ↔ tool_node` 子圖
2. `router` 從 `[[intents]]` 動態生成分類選項，將相關問題路由到新 agent
3. `Send()` fan-out 支援多意圖平行派發（含新 agent）

## 8. 新增意圖路由 (Intent)

意圖決定了使用者的問題該交給哪個 Agent。路由完全由 `config.toml` 驅動。

### 設定格式

```toml
[[intents]]
name = "warranty"                    # 意圖唯一識別名（英文）
label = "保固查詢"                    # 中文顯示名稱
description = "使用者詢問保固期限、保固範圍、保固維修"  # 描述（會注入到 router prompt）
target = "product_expert"            # 目標 agent name（必須存在於 [[agents]]）
require_slots = false                # 是否需要 slot filling
```

### 特殊意圖

| 意圖 | Target | 說明 |
|------|--------|------|
| `out_of_domain` | `out_of_domain` | 非業務問題，LLM 禮貌拒絕 |
| `transfer_human` | `human` | 使用者明確要求轉接真人 |

> `out_of_domain` 和 `transfer_human` 不會與其他意圖混合——若出現則為唯一意圖。

### Router 工作原理

`router` 節點（`graph/nodes.py`）在每次請求時：
1. 從 `[[intents]]` 動態建構意圖選項清單
2. 使用 `agents/prompts/router.md` 模板呼叫 LLM
3. LLM 回覆一或多個意圖名稱（每行一個，最多 3 個）
4. 透過 `intent → target` 對應表轉為 agent 名稱
5. 去重後設入 `state.next_agents`
6. `route_by_intent()` 使用 `Send()` fan-out 平行派發

## 9. 新增 Prompt 模板

### Prompt 存放位置

所有 prompt 統一放在 `agents/prompts/` 目錄，以 `.md` 格式管理。

### 現有 Prompt 一覽

| 檔案 | 用途 | 呼叫位置 | 模板變數 |
|------|------|---------|---------|
| `router.md` | 意圖分類 | `graph/nodes.py` → `router()` | `{domain}`, `{intent_list}` |
| `product_expert.md` | 產品規格 Agent | `agents/__init__.py` → agent 子圖 | `{domain}`, `{user_profile}`, `{slots_section}` |
| `troubleshooter.md` | 故障排除 Agent | 同上 | 同上 |

所有 Agent prompt（`product_expert.md`、`troubleshooter.md` 等）必須包含：

```markdown
3. 你必須使用工具來檢索資料，不可直接憑記憶回答。對話摘要（[前情提要]）僅供理解上下文，不可取代工具檢索——每次回覆前都必須先呼叫工具。
```

> 缺少此規則會導致 LLM 用 summary 內容直接回答而跳過 tool call。

## 10. 新增 Slot Filling 欄位

Slot Filling 用於故障排除等場景，確保 Agent 收集到必要資訊。

### 設定格式

```toml
[required_slots]
device_model = "使用者的電子鎖型號。"
device_brand = "使用者的電子鎖品牌。"
# install_date = "使用者的安裝日期。"    # ← 新增欄位只需加一行
```

### 運作原理

1. `agents/__init__.py` 的 `_build_slots_section()` 從 `[required_slots]` 自動產生提示段落
2. 透過 `{slots_section}` 注入到 Agent prompt
3. 只有 `require_slots = true` 的意圖路由所對應的 Agent 會在 prompt 中看到 slot 提示
4. Agent 會根據 `user_profile` 判斷哪些欄位已知、哪些需要詢問

## 11. 對話記憶管理設定

### 設定格式

```toml
[memory]
type = "sqlite"                    # memory | sqlite | postgres
sqlite_path = "./data/db/chat_history.db"  # sqlite 專用
max_messages_threshold = 10        # 超過此數量觸發摘要壓縮
context_retention_pair = 2         # 壓縮後保留的對話對數（1 pair = 1 user + 1 ai）
```

### 壓縮流程 (`manage_memory` 節點)

1. 每輪開始時檢查 `messages` 數量是否超過 `max_messages_threshold`
2. 超過 → 呼叫 LLM 使用 `summarize_messages.md` 摘要舊訊息
3. 透過 `RemoveMessage` API 刪除已摘要的訊息
4. 保留最近 `context_retention_pair × 2` 條訊息
5. 摘要存入 `state.summary`，由 `pre_process` 以 `[前情提要]` SystemMessage 注入

---

## 12. Send() 狀態清洗機制

Router 透過 `Send()` 將 state 派發給 agent 子圖時，會進行狀態清洗：

```python
# graph/builder.py → route_by_intent()

# 只保留 summary (SystemMessage) + 當前問題 (最後一個 HumanMessage)
agent_msgs = [系統摘要] + [當前使用者問題]

clean = {**state, "history": [], "messages": agent_msgs}
return [Send(agent_name, clean) for agent_name in valid_agents]
```

**為什麼需要清洗？**

| 問題 | 根因 | 清洗解法 |
|------|------|---------|
| `history` 每輪倍增 | `operator.add` fan-in 重複累加 | 清空 `history: []` |
| Agent 跳過工具呼叫 | 舊 AI 回覆誤導 LLM 模仿 | 過濾掉舊 AI messages |
| `merge_answers` 多算 | 掃描到舊保留的 AI 回覆 | 只傳當輪 messages |

## 13. 使用者輪廓設定

### 設定格式

```toml
[user_profile]
enabled = true
profile_dir = "./data/profiles"

[user_profile.extraction]
phone_regex   = '09\d{2}[\-\s]?\d{3}[\-\s]?\d{3}'
address_regex = '[\u4e00-\u9fff]*(?:市|縣)[\u4e00-\u9fff]*(?:區|鄉|鎮|市)[\u4e00-\u9fff\d\s\-]*(?:路|街|巷|弄|號|樓)[\u4e00-\u9fff\d\s\-]*'
```

### 運作原理

1. `pre_process` 從 `data/profiles/{user_id}.md` 載入輪廓
2. 輪廓注入到 Agent prompt 的 `{user_profile}` 變數
3. `update_profile` 節點使用 `update_profile.md` prompt 從對話中萃取新個資
4. `transfer_to_human` 節點從輪廓 + 當前訊息中 regex 提取電話、地址
5. `[user_profile.extraction]` 的 regex 由 `core/constants.py` 動態載入，支援國際化擴充

### 國際化擴充範例

若需支援日本電話格式，只需修改 `config.toml`：

```toml
[user_profile.extraction]
phone_regex = '0[0-9]{1,4}[\-\s]?[0-9]{1,4}[\-\s]?[0-9]{4}'
```

無需修改任何 Python 程式碼。
