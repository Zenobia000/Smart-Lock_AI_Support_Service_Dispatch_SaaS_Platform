# 系統部署與環境架設指南

本文件說明 Smart Lock AI Agent 系統的環境需求、變數設定、LINE Bot 串接，以及本地開發與生產環境的部署流程。

---

## 1. 系統需求

### 1.1 基礎環境

| 項目 | 需求 |
|------|------|
| Python | 3.11 以上 |
| pip | 最新版本（建議搭配 virtualenv 或 venv 使用） |
| 作業系統 | Linux / macOS / Windows（生產環境建議 Linux） |

### 1.2 依據 LLM Provider 的額外需求

| Provider | 額外需求 |
|----------|---------|
| `ollama` | 安裝 [Ollama](https://ollama.com/) 並預先拉取模型（如 `ollama pull gemma3:4b`、`ollama pull nomic-embed-text`） |
| `gemini` | 取得 Google Gemini API Key |
| `vertexai` | 安裝 [gcloud CLI](https://cloud.google.com/sdk/docs/install) 並完成 `gcloud auth application-default login` |

## 2. 環境變數完整清單

系統透過 `.env` 檔案管理機密資訊。`config.toml` 中帶有 `_env` 後綴的參數代表「環境變數名稱」，實際值由 `.env` 提供，避免機密外洩至版本控制。

| 變數名 | 必要性 | 預設值 | 說明 |
|--------|--------|--------|------|
| `LINE_CHANNEL_SECRET` | 必要 | 無 | LINE Webhook 簽章驗證 |
| `LINE_CHANNEL_ACCESS_TOKEN` | 必要 | 無 | LINE 訊息傳送（Reply / Push API） |
| `GEMINI_API_KEY` | 條件必要 | 無 | `[llm].provider = "gemini"` 時必要 |
| `VERTEX_PROJECT_ID` | 條件必要 | 無 | `[llm].provider = "vertexai"` 時必要 |
| `VERTEX_LOCATION` | 條件必要 | 無 | `[llm].provider = "vertexai"` 時必要（例: `us-central1`） |
| `OLLAMA_BASE_URL` | 選填 | `http://localhost:11434` | Ollama 服務連線網址，`config.toml` 中 `base_url` 為 fallback |
| `ORDER_API_URL` | 選填 | `config.toml` 中的 `endpoint` | 訂單查詢 API 實體位置，優先於 `endpoint` 設定值 |
| `ORDER_API_TOKEN` | 選填 | 無 | 訂單查詢 API 的 Bearer Token |
| `POSTGRES_URI` | 條件必要 | 無 | `[memory].type` 或 `[storage].type` 為 `"postgres"` 時必要 |

> **`_env` 命名慣例**：`config.toml` 中如 `api_key_env = "GEMINI_API_KEY"`，代表系統會從環境變數 `GEMINI_API_KEY` 讀取實際值。此慣例貫穿所有區塊。

## 3. LINE Bot 串接設定

### 3.1 LINE Developer Console 操作步驟

1. 前往 [LINE Developers Console](https://developers.line.biz/) 登入
2. 建立一個新的 **Messaging API Channel**
3. 在 Messaging API 頁籤中：
   - 設定 **Webhook URL** 為 `https://{your-domain}/webhook`
   - 開啟 **Use webhook**
   - 關閉 **Auto-reply messages**（自動回覆）
   - 關閉 **Greeting messages**（加好友歡迎訊息），或依需求保留
4. 在 Basic settings 頁籤複製 **Channel Secret**，填入 `.env` 的 `LINE_CHANNEL_SECRET`
5. 在 Messaging API 頁籤發行 **Channel Access Token (long-lived)**，填入 `.env` 的 `LINE_CHANNEL_ACCESS_TOKEN`

### 3.2 Loading Animation 設定

`config.toml` 的 `[line_bot].loading_animation_time` 控制使用者送出訊息後顯示「輸入中」動畫的秒數。

- 必須為 **5 的倍數**，範圍 **5 ~ 60**
- 預設值為 `60`（在系統處理期間持續顯示動畫）
- 若設定為不合法的值，LINE API 會回傳錯誤

## 4. 本地開發流程

### 4.1 安裝依賴

```bash
python -m venv venv
source venv/bin/activate    # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 4.2 建立向量資料庫

首次使用或需要重建 ChromaDB 時執行：

```bash
python scripts/seed_db.py
```

此腳本會清除並重建 `config.toml` `[[databases]]` 中所有 `type = "chroma"` 的向量資料庫。

### 4.3 Ngrok 設定（本地開發串接 LINE）

LINE Webhook 要求 HTTPS 端點，本地開發可使用 Ngrok：

```bash
# 安裝 Ngrok（https://ngrok.com/）
ngrok http 8000
```

1. 複製 Ngrok 產生的 HTTPS URL（如 `https://xxxx.ngrok-free.app`）
2. 至 LINE Developer Console 將 Webhook URL 設定為 `https://xxxx.ngrok-free.app/webhook`
3. 點擊 **Verify** 確認連線成功

> 注意：每次重啟 Ngrok 會產生新的 URL，需重新更新 LINE Console。

### 4.4 啟動服務

```bash
uvicorn app:app --reload
```

服務預設在 `http://localhost:8000` 啟動，`--reload` 會在程式碼變更時自動重啟。

### 4.5 CLI 測試

不需 LINE Bot 即可在終端機測試 LangGraph 管線：

```bash
python main.py
```

### 4.6 Mock API

若需測試訂單查詢 API（`db_order_api` 檢索器），可啟動內建的 Mock API 伺服器：

```bash
uvicorn scripts.mock_api:app --port 8001
```

搭配 `.env` 設定 `ORDER_API_URL="http://localhost:8001/v1/status"` 即可。

## 5. 生產環境部署

### 5.1 ASGI 伺服器

建議使用 Gunicorn 搭配 Uvicorn worker：

```bash
gunicorn app:app -w 1 -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000
```

**重要：worker 數量必須設為 1。** 系統的訊息防抖層（debounce）使用 in-process 字典管理計時器，多 worker 會導致同一使用者的訊息分散到不同 process，破壞防抖邏輯。

若需水平擴展，請將 debounce 改為 Redis 等外部狀態管理（需額外開發）。

### 5.2 SQLite 限制與 PostgreSQL 切換

SQLite 適合單機、低併發場景。當遇到以下情況時，建議切換至 PostgreSQL：

- 併發使用者數量持續超過 50
- 資料庫檔案超過 1GB
- 需要多機部署

切換方式：

1. 在 `.env` 設定 `POSTGRES_URI="postgresql://user:pass@host:5432/dbname"`
2. 修改 `config.toml`：
   ```toml
   [memory]
   type = "postgres"
   # postgres_uri_env = "POSTGRES_URI"   # 取消註解

   [storage]
   type = "postgres"
   # postgres_uri_env = "POSTGRES_URI"   # 取消註解
   ```

### 5.3 HTTPS 與反向代理

LINE Webhook 要求 HTTPS。生產環境建議使用 Nginx 反向代理 + SSL 憑證：

```nginx
server {
    listen 443 ssl;
    server_name your-domain.com;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

SSL 憑證可透過 [Let's Encrypt](https://letsencrypt.org/) 免費取得，搭配 Certbot 自動續期。

### 5.4 環境變數管理

生產環境不建議使用 `.env` 檔案，應改用雲端 Secret Manager：

| 平台 | 工具 |
|------|------|
| Google Cloud | Secret Manager |
| AWS | Secrets Manager / Parameter Store |
| Azure | Key Vault |

將機密值注入環境變數後，系統的 `_env` 慣例會自動讀取，無需修改程式碼。

務必確保：

- `.env` 檔案已加入 `.gitignore`，不進入版本控制
- 生產機器上移除 `--reload` 旗標
- 限制 webhook 端點的來源 IP（LINE 平台 IP 範圍）
