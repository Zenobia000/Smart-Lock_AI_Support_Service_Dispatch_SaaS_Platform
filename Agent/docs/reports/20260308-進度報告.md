# 進度報告：話題轉換偵測與主動關心機制 (2026-03-08)

## 📌 已完成項目

承接 `docs/20260307-進度報告.md` 的架構，本次完成對話記憶中「話題轉換偵測」與「主動關心舊話題」的完整實作，取代原本在 `generate_answer` 中效果不佳的「話題結束偵測」機制。

### 1. 話題轉換偵測 — 從 `rewrite_query` 階段偵測話題是否改變

- **問題描述：** 前一版在 `generate_answer` 結束後判斷「話題是否已解決」（`resolution_detection`），但實測 LLM 傾向判定 `resolved=False`（因為用戶可能會追問），導致 session 不會遞增，`chat_history` 持續累積舊話題內容，污染後續回答。
- **實作方式：**
    - **`graph/nodes.py` (`rewrite_query`)**：當有 `chat_history` 且 `topic_shift_detection = true` 時，用一次 LLM 呼叫同時做話題比對 + 改寫，回傳 JSON：`{"same_topic": bool, "previous_topic_summary": str|null, "rewritten_query": str}`。
        - `same_topic = true`：正常改寫（納入歷史上下文），行為與過去相同。
        - `same_topic = false`：忽略舊對話歷史獨立改寫，存入 `previous_topic`（前一話題摘要），history 加入 `topic_shifted` 標記。
    - **`graph/state.py`**：新增 `previous_topic: str` 欄位，用於在 `rewrite_query` 與 `generate_answer` 之間傳遞話題摘要。
    - **`config.toml`**：`[memory]` 區段移除 `resolution_detection` 與 `resolution_threshold`，新增 `topic_shift_detection = true`。
- **影響範圍：** `graph/nodes.py`、`graph/state.py`、`config.toml`

### 2. 主動關心舊話題 — `generate_answer` 附上關心語句

- **問題描述：** 使用者切換話題後，系統只回答新問題，未關心前一個問題的處理結果，服務體驗不夠貼心。
- **實作方式：**
    - **`graph/nodes.py` (`generate_answer`)**：
        - 移除整個 `resolution_detection` 區塊（LLM 判定 resolved/confidence 的邏輯）。
        - 當 `previous_topic` 有值時：
            1. 回覆的 prompt 不使用 `chat_history`（避免舊話題污染新回答）。
            2. 在回答尾巴附上關心語句：「另外，想關心一下您之前「{previous_topic}」的問題有順利解決嗎？如果還需要協助，隨時跟我說喔！」
            3. 在 history 中加入 `topic_resolved` 標記，觸發 `app.py` 的 session 遞增。
- **影響範圍：** `graph/nodes.py`

### 3. 修復 History 累加式 Bug — `app.py` 只取當次 Run 新增的 History

- **問題描述：** `GraphState.history` 使用 `Annotated[list, operator.add]` 累加式設計，MemorySaver 會保留所有歷史 run 的記錄。`app.py` 中 `"ask_missing_slots" in history` 和 `"topic_resolved" in history` 檢查的是**累積 history**，一旦某次 run 出現過這些標記，後續每次 run 都會永遠為 `True`。
    - **具體症狀：** 步驟 1 觸發 `ask_missing_slots` 後，步驟 2 完整回答了問題（slots 已填滿，走完檢索 + 生成），但因為 history 中仍殘留步驟 1 的 `ask_missing_slots`，`needs_followup` 永遠為 True → `awaiting_clarification` 被設回 True → 後續新訊息被當成反問回覆，所有訊息不斷合併，話題轉換偵測無法觸發。
- **修復方式：**
    - **`app.py` (`run_langgraph`)**：在 `ainvoke` 前透過 `aget_state` 記錄執行前 history 長度，執行後用 `full_history[prev_history_len:]` 只回傳本次 run 新增的 items。
- **影響範圍：** `app.py`

## 🔄 完整劇本預期流程

| 劇本 | 說明 | 預期路徑 |
|------|------|----------|
| 1 - Slot Filling | 對話記憶與主動追問 | `load_profile → rewrite → intent → extract → ask_missing_slots → ... → generate → update_profile` |
| 2 - 優雅降級 | 不知道型號 + 轉接真人 | `load_profile → rewrite → intent → extract → ask_missing_slots → ... → generate (含溫馨提示) → update_profile` |
| 3 - 意圖空降 | 訂單查詢直接走 API | `load_profile → rewrite → intent (order_status) → extract → db_order_api → generate → update_profile` |
| 4 - 領域護欄 | 非業務問題直接攔截 | `load_profile → rewrite → intent (out_of_domain) → extract → out_of_domain → update_profile` |
| 5 - 網頁搜尋救援 | 內部查無、網頁搜尋補救 | `load_profile → rewrite → intent → extract → db_smartlock_manual → ... → db_web_search → generate → update_profile` |
| 6 - 輪廓建立 | 首次對話萃取個人資訊 | `load_profile (空) → ... → generate → update_profile (建立)` |
| 7 - 跨 Session 記憶 | 輪廓載入提供個人化回覆 | `load_profile (載入) → rewrite (含輪廓) → ... → generate (個人化) → update_profile` |
| 8 - 轉接帶入資訊 | 自動帶入地址電話品牌 | `load_profile → ... → human (帶入已知資訊) → update_profile` |
| 9 - Slots 預填 | 從輪廓預填，免追問 | `load_profile (預填 slots) → rewrite → intent → extract → [檢索] → generate → update_profile` |
| **10 - 話題轉換** | **偵測話題切換 + 主動關心** | **`load_profile → rewrite (topic_shifted) → intent → extract → [檢索] → generate (含關心語句 + topic_resolved) → update_profile`** |
