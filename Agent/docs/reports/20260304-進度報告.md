# **會議主題：**系統架構重構、流程優化與動態記憶機制導入

### 📌 決議事項與功能規格調整

**1. 核心流程架構調整 (LangGraph Flow)**

- **現況/痛點：**原規劃的節點順序為「問題改寫 → 資訊盤點 → 意圖偵測」。
- **決議：**調整對話心智圖流程。新流程改為**「問題改寫 →  意圖偵測 → 資訊盤點」**。
- **目的：**先透過`extract_slots`釐清必填欄位（如電子鎖品牌、型號），先進行意圖分流在確保資訊齊全，以提高後續資料庫檢索與意圖判斷的精準度。

**2. 意圖與資料庫模組 Agent 化**

- **現況/痛點：**目前各意圖路由與知識庫（如`db_smartlock_manual`、`db_troubleshooting`等）為單純的檢索器串接。
- **決議：**將各別的「意圖 (Intent)」與「資料庫 (Database)」升級並封裝成獨立的 Agent。
- **目的：**賦予各模組更高的自主處理能力，為後續更複雜的任務委派與多 Agent 協作打下基礎。

**3. 複雜語意處理：多重意圖平行運算**

- **現況/痛點：**面對使用者輸入長篇大論時，現有系統可能只捕捉到單一意圖。
- **決議：**升級意圖偵測邏輯，支援「多意圖平行判斷與處理」。
- **目的：**當使用者同時詢問（例如：「我的 X1 Pro 沒電了怎麼辦？另外我上週買的配件出貨了嗎？」），系統能平行觸發「故障排除 Agent」與「訂單查詢 Agent」進行處理。

**4. 觸發回覆機制優化 (導入 LLM 信心指數)**

- **現況/痛點：**原本依賴前端或中介層「等待最後一次訊息後 5 秒」作為對話終止與回覆的觸發條件，體驗較生硬且缺乏彈性。
- **決議：**廢除 5 秒等待機制。改用語意驅動，利用 LLM 評估「信心指數 (Confidence Score)」。
- **目的：**由 LLM 實時判斷目前收集到的歷史對話與槽位資訊是否「足以回答該意圖」。若信心指數達標即刻觸發生成回覆；若不足則繼續等待或主動反問。

**5. 個人化記憶機制：動態使用者輪廓 (User Profile)**

- **現況/痛點：**缺乏跨工作階段 (Session) 的長期使用者背景記憶。
- **決議：**導入基於`.md`檔案格式的動態記憶機制。
    - **記錄：**根據使用者過往對話，生成並萃取個人化資訊（如：住家環境、慣用設備型號等）。
    - **更新：**每次對話結束後，由系統判斷是否需要更新該使用者的`.md`輪廓檔案。
    - **應用：**在每次啟動對話時，將此`.md`檔案內容作為 Background Context 交給 LLM，提供更具脈絡的個人化客服體驗。
