# 進度報告：Slot Filling 條件化與領域護欄優化 (2026-03-06)

## 📌 已完成項目

承接 `docs/20260305-開發交接文件.md` 的架構，本次針對五劇本全面測試後發現的三個流程缺陷進行修復。

### 1. Slot Filling 條件化 — 依意圖決定是否要求必填欄位

- **問題描述：**`required_slots`（品牌/型號）對所有意圖一律強制生效，導致劇本 3（訂單查詢）、劇本 4（領域護欄）、劇本 5（網頁搜尋救援）全部卡在 `ask_missing_slots` 就結束，後續檢索流程完全無法觸發。例如使用者問「幫我查訂單進度」，系統卻反問「請問您的電子鎖型號是什麼」。
- **修復方式：**
    - **`config.toml`**：各 `[[intents]]` 新增 `require_slots` 欄位，僅 `troubleshooting = true`，其餘皆為 `false`。
    - **`graph/builder.py`**：`route_after_extract` 改為查表 `intent_require_slots`，只有標記 `require_slots = true` 的意圖才進行必填欄位檢查，其他意圖直接放行至對應的檢索節點。
- **影響範圍：**`config.toml`、`graph/builder.py`

### 2. 領域護欄前置化 — 新增 `out_of_domain` 意圖節點

- **問題描述：**使用者提出與業務完全無關的問題（如「今天台北天氣如何？」），系統會依序跑完所有 Retriever（手冊 → 故障庫 → API → 網頁搜尋），全部被 Grader 判定 NO 後才落入 `human` 節點，回覆不相關的轉接真人罐頭訊息。既浪費運算資源，使用者體驗也差。
- **修復方式：**
    - **`config.toml`**：新增 `[[intents]]` `out_of_domain`，描述為「使用者的問題明顯與電子鎖、智慧門鎖完全無關」。
    - **`graph/nodes.py`**：新增 `out_of_domain` 節點函數，從 `SYSTEM_CONFIG` 動態讀取 `domain` 欄位，回覆「這個問題超出我的服務範圍」。
    - **`graph/builder.py`**：註冊 `out_of_domain` 節點並接上 `END` 邊。
- **影響範圍：**`config.toml`、`graph/nodes.py`、`graph/builder.py`

### 3. Web Search 攔截邏輯修正

- **問題描述：**原本 `create_fallback_router` 中的 web search 攔截只在 `intent == "troubleshooting"` 時生效。當意圖為 `order_status` 且 slots 為 UNKNOWN 時，系統仍然會進入無意義的網頁搜尋。
- **修復方式：**移除 `and current_intent == "troubleshooting"` 條件，改為只要任一 slot 值為 `UNKNOWN` 即攔截，不再限定特定意圖。
- **影響範圍：**`graph/builder.py`
