根據 `docs/20260304-進度報告.md` 的規劃，目前**第一階段：核心流程架構調整 (LangGraph Flow)** 已經順利完成，並且修復了轉接真人（`transfer_human`）的 Routing Bug。目前的測試案例 (demo_1 ~ demo_5) 皆已在新架構下順利跑通。

### 1. 核心流程調整完成
*   **舊流程**：問題改寫 (rewrite) → 資訊盤點 (extract) → 意圖偵測 (intent)
*   **新流程**：問題改寫 (rewrite) → 意圖偵測 (intent) → 資訊盤點 (extract)
*   **變更檔案**：
    *   `graph/builder.py`: 修改了邊緣 (`Edges`) 的連接順序。
    *   `graph/nodes.py`: 無重大邏輯修改，但節點的觸發順序已配合 `builder.py` 變更。

### 2. 轉接真人 (transfer_human) Routing Bug 修復
*   **問題描述**：在第一階段重構後，使用者如果在未提供品牌/型號 (`REQUIRED_SLOTS`) 的情況下，明確要求轉接真人 (`transfer_human` 意圖)，系統會因為 `route_after_extract` 路由器的阻擋，陷入不斷「反問型號 (`ask_missing_slots`)」的死迴圈，無法順利轉接給真人。
*   **修復方式**：修改 `graph/builder.py` 中的 `route_after_extract` 函式，加入了「意圖豁免」邏輯。
    *   **修改內容**：
        ```python
        # 位於 graph/builder.py
        def route_after_extract(state: GraphState):
            current_intent = state.get("intent")
            target = intent_mapping.get(current_intent)
            
            # 【修復重點】如果意圖是直接轉接真人，則不需要檢查必填欄位 (Slots)，直接放行
            if target == "human":
                return current_intent

            current_slots = state.get("slots", {})
            
            for key in REQUIRED_SLOTS.keys():
                val = current_slots.get(key)
                if not val:  
                    return "ask_missing_slots"
                    
            return current_intent
        ```
*   **測試結果**：在 `demo_2` 劇本中，系統現在能夠成功跳過型號反問，直接到達 `human` 節點。

## 🚀 下一步開發建議 (交接給下一個 AI Assistant)

目前的程式碼已經為接下來的「Agent 化」打下了穩固的基礎。請接手的開發者從**第二階段**繼續進行。

### 任務：意圖與資料庫模組 Agent 化

目前的架構還是依賴寫死的 Fallback 判斷 (`create_fallback_router`)，這不符合未來多 Agent 協作的需求。接下來需要將其重構為基於 Tool Calling 的 Agent 迴圈。

**推薦實作步驟：**

1.  **升級 State (狀態)**
    *   **目標檔案**: `graph/state.py`
    *   **動作**: 將 `GraphState` 升級為支援 Agent 的狀態。加入 `messages` 欄位（`Annotated[Sequence[BaseMessage], operator.add]`），以容納 LangChain 原生的 Tool Messages。

2.  **工具化 (Toolify) 檢索器**
    *   **目標檔案**: 新增 `graph/tools.py` 或修改現有 `retrievers/`。
    *   **動作**: 將 `config.toml` 中定義的各個資料庫（產品手冊、故障排除、訂單 API、網頁搜尋）封裝成具備 `@tool` 裝飾器的 LangChain 工具。並為每個工具寫上清晰的 docstring，說明該工具的使用時機。

3.  **引入 Agent 節點**
    *   **目標檔案**: `graph/nodes.py`
    *   **動作**: 建立一個具有呼叫工具能力的 `customer_service_agent` 節點。這個節點會讀取 `messages` 與 `slots`，決定是要直接回答，還是要呼叫剛才寫好的 Tools 進行查詢。

4.  **重構 LangGraph 圖表 (核心)**
    *   **目標檔案**: `graph/builder.py`
    *   **動作**:
        *   移除舊有的、寫死的意圖路由與 `Fallback Chain` (`create_fallback_router` 與 `decide_sufficiency`)。
        *   將流程改為：`問題改寫` -> `意圖偵測` -> `資訊盤點` -> **`Agent 節點` <-> `Tool 節點`** -> `生成回覆` (或由 Agent 直接生成)。
        *   這一步是重構的關鍵，要確保 Agent 能夠自主決定查詢哪個資料庫，並在查無資料時自主決定下一步（例如查網頁或轉真人）。
